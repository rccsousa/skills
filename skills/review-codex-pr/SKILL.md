# review-codex-pr — Grumpy PR Review

Deeply review a pull request that was (typically) generated by an AI agent such as Codex. Be **extra attentive, maximally grumpy, and relentlessly nitpicky**. The assumption is that AI-generated PRs ship with plausible-looking code that hides real gaps. Your job: find them.

## Arguments

- `$1` — PR number (required). Repo is auto-detected from `git remote get-url origin`.
- Optional: repo slug `owner/name` if the user wants a different repo than the current one.

If no number is provided, ask the user for one before proceeding.

## Process

### 1. Fetch PR context (parallel calls)

- `gh pr view <num> --repo <slug> --json title,body,author,files,additions,deletions,baseRefName,headRefName,url,state`
- `gh pr diff <num> --repo <slug>` — full diff
- If diff references files you need full context for, `Read` them directly (not just the hunk).

### 2. Review with maximum rigor

Hunt specifically for things AI-generated PRs tend to get wrong:

- **Security**: auth bypass, missing signature/timestamp/replay checks, SSRF, unbounded fetch without timeout, unbounded caches, soft-fail where prod should fail-closed, regex too permissive, env-var fallbacks that silently disable protections.
- **Crypto**: algorithm downgrade, accepting unknown versions, canonicalization mistakes, mis-ordered signed fields, trusting attacker-controlled URLs after signature, key-material handling.
- **Correctness**: off-by-one, unhandled enum variants / `Type` branches falling through, optional fields not covered, type assertions hiding `undefined`.
- **Nest/Fastify specifics**: undecorated controller method params, route handlers called as plain functions from other handlers, header types lying (`Record<string, string>` when values can be `string[] | undefined`), `@Body()` expecting raw string but no `rawBody` config, interceptors not applied to new routes.
- **Tests**: happy-path-only; missing negative, boundary, malformed-input, and cache/TTL behavior; over-mocking; tests that assert nothing meaningful.
- **Design smells**: dual-use functions, generic classes coupled to specific consumers via direct `env` access, over-engineered type generics, dead exports.
- **Docs / PR body**: env vars undocumented, default behavior not called out, risk not acknowledged.
- **Logging**: per-request `log` level noise that should be `debug`; logs leaking secrets or tokens; missing structured fields.
- **Tiny nits**: redundant type annotations, `any` escape hatches in tests, stale comments, dropped TODOs, bad variable names, useless try/catch that swallows.

Always **cite file:line** and include doc links for standards-based claims (AWS, RFC, etc.) — this matches the user's global "cite sources" rule.

### 3. Structure the review

Apply the shared **review output contract** at `~/.claude/lib/review-output-contract.md` — severity buckets (Critical / Major / Minor / Strengths / Followup risks), file:line citing, verdict format, DO/DON'T rules.

Tone-overlay for this skill: grumpy but specific. No pointless shaming — every complaint must be actionable with a suggested fix or acceptance criterion. Test-gap findings go under Major (or Minor if happy-path-only on a low-risk path).

### 4. Draft — DO NOT POST

Write the review body to `/tmp/pr<num>-review.md`. **Stop there.** Do NOT call `gh pr review`, `gh pr comment`, `gh api`, or anything that writes to GitHub. Submitted reviews cannot be deleted — only the body can be edited after the fact. Posting without explicit approval is a trust violation.

Show the user:
- A short summary of findings per severity bucket.
- The path to the draft file.
- Offer: "Post as comment? Request changes? Edit first? Or just keep it local?"

Only run `gh pr review ...` after the user explicitly says post/submit/ship. If the PR is the user's own, `--request-changes` is rejected by GitHub — use `--comment` and mention that fallback before running.

### 5. Confirm

After (and only after) the user approves and you post, confirm with:

```bash
gh pr view <num> --repo <slug> --json reviews --jq '.reviews[-1] | {state, author: .author.login}'
```

## Constraints

- Respect user memory rules: no auto-commits, no touching unstaged files, cite sources with file:line.
- Do not create new files in the repo being reviewed. The review lives on GitHub.
- Do not approve (`--approve`). This skill's entire purpose is adversarial review.
- If the PR has zero real issues after honest scrutiny, say so explicitly in the Verdict — don't manufacture nits to look thorough.
